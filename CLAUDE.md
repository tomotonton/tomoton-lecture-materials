## 公開フロー

```
1. HTMLファイル作成（拡張子は必ず .html）
        ↓
2. node tools/generate-index.mjs を実行（index 再生成）
        ↓
3. SourceTree で確認 → commit
        ↓
4. main ブランチへ push
        ↓
5. Cloudflare Pages が自動デプロイ
        ↓
6. シークレットウィンドウ or ?v=YYYYMMDD でキャッシュ回避して確認
```

### よくあるトラブル

- **C++フォルダ等が空に見える** → `.md` のまま置いている／index 再生成し忘れ
- **push したのに変わらない** → Cloudflare が最新コミットをデプロイできていない／ブラウザキャッシュ
- **再実行したのに古いまま** → 再実行は同じコミットを再ビルドするだけ。新コミットを push し直す

| `sangiStats/{id}/attempted/{uid}` ・ `.../cleared/{lang}/{uid}` | サンギ事前トレーニングの回答者数・**言語別クリア人数**（`lang`=c/cpp/java。総クリア=言語をまたいだuidの和集合、`id`=`sangi-NN`）|

### クイズの仕組み

- 各HTML内の `.quiz-choices[data-qid]` が `quiz-firebase.js` によって初期化される
- 学生の回答は localStorage と Firebase の両方に保存
- 過去48時間の集計を、**選択肢ブロックの右側の専用領域に横棒グラフ**（選択肢ごと・得票数・%・色分け：正解=緑/自分の誤答=赤/その他=青）でリアルタイム表示。回答後に右領域が出現し、狭い画面では選択肢の下に回り込む（自前描画・Chart.js不使用。選択肢ボタン自体は従来通り）
- 管理者は右下の「データリセット」ボタン（キーワード `ignas`）でクリア可能

### Firebase セキュリティルール（匿名認証・2026-06-14 適用済み）

「安全でないルールがあります」警告メール（＝`.read/.write: true` の完全公開への通知）を**根本的に止める**ため、全パス公開をやめ、**匿名認証（Anonymous Auth）＋ `auth != null`** に移行**済み**。クライアントは全ページで**学生に見えない匿名サインイン**を自動実行する（学生は何の操作もログイン画面も不要・体験は不変）。現行ルールは次の通り：

```json
{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null"
  }
}
```

- **匿名サインインを実装済みのファイル**: `quiz-firebase.js`（×4: アルゴリズム/C言語/C++(CPP)/Java）・`C言語練習問題/c-judge.js`・`C++練習問題/cpp-judge.js`・`C言語/03-おまけ_2進-16進変換トレーニング.html`。いずれも Firebase 初期化直後に `signInAnonymously` し、その完了（成功/失敗どちらでも解決する `authReady` という Promise）を待ってから読み書きする。**サインインが失敗しても握りつぶして続行**するので、ルールが公開のままでも従来どおり動く（＝無停止で移行できる）。
- **有効化の経緯（実施済み・この順番。各ステップは独立して安全）**:
  1. コードを push（Cloudflare 自動デプロイ）。この時点ではまだ公開ルールのままなので全機能そのまま。
  2. Firebase コンソール → Authentication → Sign-in method → **「匿名」を有効化**。これで各ページの匿名サインインが成功し始める（それまでコンソールに出る `auth/configuration-not-found` 警告が消える）。
  3. Realtime Database → ルール を上記 `auth != null` に変更して公開。→ **これで「安全でないルール」警告メールが止まる。**
- ✅ **2026-06-14 に①②とも実施・動作確認済み**（匿名サインイン成功＋ `auth != null` 下で本番データの読み書きOKを実機確認）。
- ⚠ コンソールに残る黄色バナー「認証されたユーザーであれば誰でも…」は `auth != null` だと出る**想定どおりの注意書き**で、**警告メールのトリガーではない**（出たままで正常・無視でよい）。
- ⚠ ルール変更直後、**変更前に開かれていた古いタブ**は匿名トークンを持たず書き込みに失敗する（リロードで回復）。新規アクセスは問題なし。
- ⚠ 匿名認証は誰でもトークンを取得できるため、これだけでは不正投稿を完全には防げない（教室用途では許容）。厳密に塞ぐならパスごとの検証ルールを追加する。
- `answers`/`meta`（クイズ）・`bhScores`（2進16進）・`practiceStats`（練習問題）すべてこの1つのルールでカバー。**新しいデータパスを増やしてもルール変更は不要**。
- 管理リセットは各画面の右下⚙（キーワード `ignas`）または Firebase コンソールから（従来どおり）。

# 個別仕様

## 2進数・16進数変換トレーニング

### ファイル

`C言語/03-おまけ_2進-16進変換トレーニング.html`

### 背景

- 30秒・0〜F・両方向ランダム のみの初期版は難しすぎた
- 学生（A26043 望月啓希さん）から「制限時間が短く考えてる間にゲームオーバー・暗記ゲーになる」とフィードバック
- 範囲を狭める／時間を伸ばす／方向を片方向に絞ることでステップアップ式に再設計

### 構成

#### トップレベル: 出題方向（2つの独立ゲーム）

| ID | 方向 | 答え |
| --- | --- | --- |
| `b2h` | 2進数 → 16進数 | 16進1桁 |
| `h2b` | 16進数 → 2進数 | 2進4桁 |

#### 各方向内のレベル（4段階）

| ID | レベル | 時間 | 範囲 | 備考 |
| --- | --- | --- | --- | --- |
| `practice` | 📚 練習 | なし | 0〜7 | 変換表ボタンあり |
| `lower` | 🌱 しただけ | 60秒 | 0〜7 | 2進数の先頭ビットが0の組 |
| `upper` | 🌿 うえだけ | 60秒 | 8〜F | 2進数の先頭ビットが1の組（A〜F含む） |
| `hard` | 🔥 チャレンジ | 60秒 | 0〜F | **ランキング対象**（全16種類） |

> 「うえだけ」を分けたのは、A〜Fの文字対応＋2進数で `1xxx` の処理という二重の難しさが集中する苦手ポイントだから。

### 主要機能

- **スマートキーパッド**: 2進答えなら `0` `1` `←` の3キー、16進答えなら `0〜F` `←` の17キー
- **自動判定**: 必要桁数（2進=4桁／16進=1桁）入力で即判定。「判定」ボタン押下不要
- **ハンドルネーム入力**: ランキング登録に使用（最大12文字、空欄ならランキング非登録）
- **過去24時間履歴**: レベル別に正解数・回答数・正答率を localStorage で記録

### ランキング仕様

- **対象は チャレンジレベル のみ**（練習・しただけ・うえだけはランキングなし）
- **左右2パネル並列表示**：
  - 左: 2進数 → 16進数 のランキング
  - 右: 16進数 → 2進数 のランキング
  - 両方向を**同時に**購読・表示 → 両方の自分の位置が一目で分かる
  - 現在プレイ中の方向のパネルは青枠でハイライト
- **得点分布ヒストグラム** で表示（順位表ではなく分布グラフ）
  - X軸: スコア（0〜最高点）
  - Y軸: その点数の人数（バーの高さ）
  - 自分の自己ベスト位置は**橙色＋▲マーカー**で強調
- **歴代TOP10ランキング表**（順位・ハンドル・点・ひとことコメント）も併記
- パネル上部に「あなたのベスト N 問 / プレイ M 回 / 全 K 人中 X 位」を表示
- **歴代（全期間）** の記録を集計（記録は消えずにずっと残る）
- 同じユーザー（同 uid）の重複は最高スコアのみ採用
- 順位は同点同順位（より高い点数の人数 + 1）
- モバイル（max-width: 520px）では2パネル縦積み

### 上位ランクイン演出（お祝い・ひとこと）

- チャレンジ終了時、**自己ベスト更新 かつ 歴代TOP10入り** で全画面お祝い演出（canvas花火＋祝福カード）
- 歴代1位（他全員を上回る新記録）は「🎆 世界新記録 🎆」、それ以外は「🎉 歴代 N 位 ランクイン！」
- 演出内で**ひとことコメント**（任意・最大30文字）を入力でき、`update()` でそのレコードに追記 → ランキング表に反映
- 発火判定は `computeRank(mode, newScore)`（送信前の `lbCache` から各uidベストを集計し `rank`/`isImprovement`/`isWorldRecord` を算出）
- 花火は外部ライブラリ不使用の自前canvasパーティクル（`startFireworks`）

### Firebase 購読パターン

- `currentLevel === "hard"` に切り替わったとき、`bhScores/b2h_hard` と `bhScores/h2b_hard` の両方を購読
- 方向タブ切り替え（`currentMode` 変更）では**再購読しない**。アクティブパネルのハイライトと「歴代自己ベスト」カードの値だけ更新
- 非チャレンジレベルへ切り替えるとき購読解除

### Firebase データ構造

```
bhScores/
  b2h_hard/{pushKey}: { handle, score, ts, uid, comment? }   ← 2進→16進 チャレンジ
  h2b_hard/{pushKey}: { handle, score, ts, uid, comment? }   ← 16進→2進 チャレンジ
```

| フィールド | 内容 |
| --- | --- |
| `handle` | ハンドルネーム（最大12文字、trim済み）|
| `score` | 正解数（整数）|
| `ts` | 投稿時刻（ミリ秒タイムスタンプ）|
| `uid` | 匿名ユーザーID（localStorage、自分の記録識別用）|
| `comment` | ひとことコメント（任意・最大30文字。上位ランクイン演出で入力時のみ付与）|

### localStorage キー

| キー | 内容 |
| --- | --- |
| `bhUserId` | 匿名ユーザーID（初回生成、永続）|
| `bhHandle` | ハンドルネーム |
| `bhHist_{level}` | 過去24時間の回答履歴（レベルごと、`{ t, ok }[]`）|

> ランキング系は Firebase が正本。localStorage は使わない。
> 個人ベスト表示も Firebase の自分のレコードから算出。

### 設計判断の経緯

1. 最初は30秒・0〜F・両方向ランダムのみ → 難しすぎた
2. **練習・かんたん・ふつう・チャレンジ** の4レベル制を導入（範囲と時間を段階化）
3. 「両方向ミックス」も検討したが、**方向判別のワンクッション**が「考えてる間に時間切れ」の主因と判断し、片方向のみに
4. 当初は過去10日窓だったが「歴代を残してさらに上を目指したい」との要望で**全期間（歴代）保存**に変更。上位ランクインは花火＋ひとことコメントで祝う
5. 練習・かんたん・ふつうは個人練習用、競争はチャレンジに集約 → ランキングは hard レベルのみ

### Firebase セキュリティルール

`bhScores` パスは read/write 許可が必要。**2026-06 に匿名認証＋ `auth != null` へ移行**（→「Firebase 連携」内の「Firebase セキュリティルール（匿名認証へ移行）」を参照）。このHTMLも匿名サインイン実装済みなので、ルールを `auth != null` にしても動く。`bhScores` 個別の追加設定は不要。

不正なスコア投稿のリスクはありますが、教室内の用途なので許容範囲。
管理者によるデータリセットは Firebase コンソール、または各画面の右下⚙（キーワード `ignas`）から実行する想定。

## サンギハッカソン対策（インタラクティブ練習問題・Wandbox採点）

`SANGI Hack Challenge 事前トレーニング/`（旧名 `sangiハッカソン対策/`）… サンギハッカソン（本番採点は Exercode）の**予習用インタラクティブ練習問題**。1問=1HTMLで、問題文＋コードエディタ＋実行＋全テスト採点まで画面内で完結する（既存の C/C++練習問題と同じ操作感だが、実コンパイラで C/C++/Java の3言語に対応）。本番の編集/コンパイル/実行/採点は Exercode が担い、ここは事前学習用。先頭に `00_SANGI Hack Challenge攻略情報.html`（採点ルール・得点分布・攻略方針をまとめた“これなら戦える”モチベ資料。データは Downloads の `SANGI攻略_対策資料.md` 等が出典）を置く。

### 構成
- 各問 `NN_タイトル.html`: 静的な問題文/制約/入出力例/学習アドバイス ＋ `<div id="judge">` ＋ `window.PROBLEM`（各言語の `starter`/`model`/`compiler` とテスト配列）＋ `<script src="sangi-judge.js">`。
- `sangi-judge.js`（共通エンジン）: エディタ＋C/C++/Javaタブ＋「コンパイル・実行」「提出（全テスト採点）」「最初のコードに戻す」。学生コードは localStorage `sangiCode:<id>:<lang>` に自動保存。
- 1セット12問（LV1〜5、各6テスト）。

### 採点エンジン（重要）
- コンパイル/実行は **Wandbox API**（`https://wandbox.org/api/compile.json`）を**ブラウザから直接** POST。Wandbox は CORS 許可済みなので**バックエンド不要**（Cloudflare Functions 等いらない）。コンパイラ: C=`gcc-13.2.0-c` / C++=`gcc-13.2.0` / Java=`openjdk-jdk-22+36`。
- **Java の落とし穴**: Wandbox はソースを `prog.java` 保存するため `public class Main` がコンパイルエラー → 送信前に `public class Main`→`class Main` へ自動変換するシムを `sangi-judge.js` が持つ（学生は普通に `public class Main` と書ける）。
- 出力比較は単語単位（空白区切り・浮動小数点は誤差許容）。Wandbox は混雑時に一時エラー（"Resource temporarily unavailable" / "OCI runtime" 等）を返すことがあるため、`sangi-judge.js` は最大2回**自動リトライ**する。
- ⚠ Wandbox は無料の共有サービス。多人数同時では遅延/一時失敗があり得る（丁寧なエラー表示＋リトライで対応。最終的な採点は Exercode 本番）。

### クリア（👑）バッジ
- 全テスト合格で「クリア」＝ localStorage `sangiClear:<正規化パス>` に保存し、親フレーム（目次SPA）へ `postMessage({type:"sangiCleared", path})`。
- ルート目次（`tools/generate-index.mjs` 生成）が `sangiClear:` を読み該当問題に 👑 を表示（C/C++練習問題の ✅ `cjPass:` とは**別系統**）。`generate-index.mjs` の `applyCrownBadges` / `sangiCleared`・`sangiUncleared` ハンドラ参照。バックスラッシュ正規表現はテンプレートで消えるため使わない（`normPath` の charAt/slice 流用）。
- 各問題ページには「👑 クリア N人（C a・C++ b・Java c）／挑戦 M人」を**リアルタイム表示**（Firebase `sangiStats`・上表参照。`sangi-judge.js` が提出時に記録＋onValue購読。匿名認証＋`auth!=null`でカバー）。管理リセットは Firebase コンソールで `sangiStats` ノード削除（専用⚙ボタンは未実装）。
- 旧フォルダ名 `sangiハッカソン対策/` でブックマーク/キャッシュした `?p=` 古いURLは、目次SPAの `remapLegacyPath` が新フォルダ名へ自動変換する。

> 2026-06: 当初 Exercode アップロード用フォルダ `sangi_practice/`（problem.md/test_cases 形式）も作ったが、「サイト上で予習できる形がよい」との要望で**本インタラクティブ版に一本化**し `sangi_practice/` は廃止。問題の素材（雛形/模範解答/テスト）は各 HTML の `window.PROBLEM` 内に保持。

## C言語資料の方針（純粋なC / C89）

- **変数宣言は必ず関数の先頭でまとめて行う**（C89スタイル）。`for (int i …)` のようなループ内宣言や、実行文（printf/scanf/代入など）の後での宣言はしない。ループ変数は `int i;` を先頭で宣言してから `for (i = 0; …)` とする。
- 変数の単元（`02_C言語の基礎_変数とデータ型.html` の 02-3）に、このルールの説明（良い例／悪い例）を記載済み。
- 改行は資料の慣習で **円記号 `¥n`（U+00A5）** で表記（バックスラッシュ `\n` ではない）。日本語Windows向けの既存仕様なので、勝手に `\n` へ置換しないこと。
- 新規作成・改修するC言語のコード例も、上記に必ず合わせること。
